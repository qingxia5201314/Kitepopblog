# Kitepop 记账模块设计规格

## 目标

在 Kitepop 站点中新增一个独立的私密记账模块。它与阅读、后台并列为顶层入口，但所有记账数据必须鉴权后才能访问。登录状态保持 30 天，过期后自动回到登录面板。

## 范围

- 顶部导航新增“记账”入口。
- 未登录记账页只显示登录面板，不展示金额、流水、目标或统计摘要。
- 记账登录复用现有后台口令校验，但使用独立会话 token。
- 记账会话有效期 30 天，浏览器保存 token，服务端 SQLite 保存 token 哈希和过期时间。
- 支持新增、编辑、删除记账流水。
- 流水字段包括类型、金额、分类、日期、账户、备注。
- 支持按月份、类型、分类筛选流水。
- 支持月度统计：收入、支出、结余、日均支出、最大支出分类。
- 支持月预算：预算金额、已用比例、剩余金额。
- 支持一个月存钱目标：目标金额、开始日期、结束日期、当前已存、进度、剩余金额、每日建议存入金额。
- 记账数据写入服务端 SQLite，与博客文章共用数据库文件但使用独立表。

## 非目标

- 不做多用户系统。
- 不做云同步到第三方财务平台。
- 不做自动导入账单、OCR、银行卡连接。
- 不在 Git 中提交真实记账数据、真实 token、后台口令或 VPS 凭据。

## 鉴权与安全

记账接口统一挂在 `/api/accounting` 下，除登录接口外均要求 `Authorization: Bearer <token>`。服务端只保存 token 的 SHA-256 哈希、创建时间和过期时间，不保存明文 token。浏览器使用 `localStorage` 保存记账 token 和过期时间，用于 30 天会话保持；服务端仍是最终裁决，过期或无效 token 会返回 401。

记账会话与文章后台会话分离，避免后台发布状态和记账状态互相影响。两者可以复用同一个 `ADMIN_PASSWORD` 做口令校验，但 token 命名、存储和接口权限独立。

## 数据模型

SQLite 新增三类表：

- `accounting_sessions`：保存记账登录 token 哈希、创建时间、过期时间。
- `accounting_entries`：保存收入/支出流水。
- `accounting_settings`：保存月预算和一个月存钱目标配置。

金额统一以“分”为单位保存，前端输入和展示使用元，避免浮点误差。日期使用 `YYYY-MM-DD` 字符串，月份筛选使用 `YYYY-MM`。

## 前端设计

顶栏变为“阅读 / 记账 / 后台”。记账页登录后使用和当前站点一致的浅色背景、深色按钮、紧凑卡片与分段控件。

记账主界面包含：

- 顶部月度概览条：收入、支出、结余、预算剩余。
- 快速记一笔表单：类型、金额、分类、日期、账户、备注。
- 筛选工具条：月份、类型、分类。
- 流水列表：可编辑、删除。
- 存钱目标面板：目标金额、周期、当前进度、每日建议。

## 验证

- 服务端存储测试覆盖会话过期、流水 CRUD、月度统计、预算与目标计算。
- 前端 API 测试覆盖鉴权 header、登录 token、流水与设置请求。
- 构建前运行 `npm test -- --run` 和 `npm run build`。
- 部署后验证线上首页、记账登录接口、鉴权失败返回 401、文章接口仍正常。
